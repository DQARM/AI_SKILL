# ref2pptx — References → PPTX(參考資料轉投影片產生器)

把一個**專案資料夾**裡的參考資料,轉成一份**對指定受眾真正聽得懂**的 pptx 投影片,並把產出寫回專案資料夾的 `30_Output/`。

這個 skill 的價值不在於「會做投影片」(實際排版交給 `pptx` skill),而在兩件事:一是**先把素材篩過一輪**(採用 / 排除分流,讓做簡報的資料是乾淨的);二是**強制每個章節都通過受眾人物誌審查**——用一位真實背景的受眾的眼睛重讀章節,確認沒有他看不懂的術語、且有足夠且貼近他日常工作的例子。這是為了避免簡報變成「作者覺得清楚、受眾一臉問號」。

---

## 專案資料夾結構

這個 skill 以**一個專案資料夾**為單位運作。使用者會給你專案資料夾的路徑,裡面用**編號子資料夾**分區:

```
<專案資料夾>/
├── INDEX.md         專案總覽地圖(skill 自動維護,給人快速掌握全局)
├── README.md        專案說明:這是什麼、怎麼用 / 重跑(skill 自動維護,給人看)
├── CLAUDE.md        專案記憶:給 Claude Code 的慣例與現況(skill 自動維護,給 AI 看)
├── AUDIENCE.md      受眾設定(控制檔,放專案根目錄)
├── OUTLINE.md       章節結構 + 各章內容(投影片主來源,放專案根目錄)
├── 00_Masters/      投影片母片 / 範本(.pptx / .potx,選用)
├── 10_Input/        資料暫存區:新素材先丟這裡,待篩選
├── 20_References/   補充參考素材(圖、要引用的文件等;skill 會一併取用)
├── 30_Output/       產出:簡報 .pptx + review-report.md
└── 90_Excluded/     評估過、不適合使用的資料(連同排除原因)
```

`AUDIENCE.md`、`OUTLINE.md` 是**控制檔,放在專案根目錄**(不放進 `20_References/`)。其中 **`OUTLINE.md` 同時是章節結構與各章內容,是投影片的主來源**——調章節順序、改內容都在這個頂層檔完成,不必進子資料夾。控制檔不參與篩選、永遠不會被移到 `90_Excluded/`。

**主來源 vs 補充素材**:投影片骨架與內容以根目錄 `OUTLINE.md` 為主;`20_References/` 是**補充參考素材**(圖、要引用的文件、數據來源等),skill 會取用來豐富 / 佐證各章,但不是骨架。兩者衝突時以 `OUTLINE.md` 為準。章節順序一律以 `OUTLINE.md` 的標題順序為準,skill 不自行重排。

`INDEX.md`、`README.md`、`CLAUDE.md` 是 **skill 在每次跑完自動寫 / 刷新的根目錄文件**(見第 10 步),三者用途不同、不要重複:`INDEX.md` 是當前狀態地圖、`README.md` 給人看怎麼用、`CLAUDE.md` 給 Claude Code 記住專案慣例與現況。它們都不是素材。

開工前先用 `view` 列出專案資料夾,確認各區狀態:

- `20_References/` 和 `10_Input/` **兩個都空 / 都不存在**:沒有任何素材可用,停下來問使用者素材放哪,不要自己亂猜。
- `30_Output/` 不存在:直接建立一個再往下。
- `00_Masters/`、`90_Excluded/` 不存在:需要時自動建立即可(`90_Excluded/` 在第一次排除素材時建)。

子資料夾名稱以上述編號為預設;若使用者的專案用別的名字,依使用者實際命名為準,但**角色對應不變**(母片 / 暫存 / 採用 / 產出 / 排除)。

### 00_Masters — 投影片母片

放 PowerPoint 範本檔(`.pptx` / `.potx`)。**選用**。處理規則見下方「開工前必問 → 母片」:只要裡面有檔案就**必須先問使用者**,不自作主張。

### 10_Input — 資料暫存區

使用者把**新蒐集、還沒篩過**的素材先丟這裡(可以是單檔,也可以是**整個資料夾**)。skill 會在第 1 步把這裡的東西逐件評估,分流到 `20_References/`(採用)或 `90_Excluded/`(不適用);**資料夾以整個為單位搬移**(見第 1 步)。處理完後 `10_Input/` 理想上會被清空(東西都分流走了)。

### 20_References — 補充參考素材

放**補充用的參考素材**,用來豐富 / 佐證各章內容(投影片骨架與主內容來自根目錄 `OUTLINE.md`)。可以是使用者直接放好的、也可以是從 `10_Input/` 篩選後移進來的。可有可無——只有 `OUTLINE.md` 也能產出簡報。

| 內容 | 用途 | 是否必要 |
|------|------|----------|
| 圖、截圖、流程圖 | 放進對應章節的視覺 | 選用 |
| 要引用的文件 / 數據來源(md / docx / pdf / txt) | 佐證、補充細節 | 選用 |
| 含內容的資料夾(可含 `URL.txt`、`(O)` 記號) | 成組的素材 + 來源網址 | 選用 |
| 品牌 / 風格說明、Logo、配色 | 套用視覺風格 | 選用 |

> 主來源是根目錄 `OUTLINE.md`;`20_References/` 是補充。兩者衝突以 `OUTLINE.md` 為準。設定檔 `AUDIENCE.md`、`OUTLINE.md` 放根目錄,不放這裡。

讀檔前先用 `file-reading` skill 判斷各檔該怎麼讀(尤其 pdf / docx / 圖,不要直接 `cat` 二進位檔)。

### 30_Output — 產出

所有產出都寫進這裡,**不要寫到別區,也不要改動 `20_References/`、`00_Masters/`、`90_Excluded/`**:

- `<簡報名>.pptx` —— 最終簡報(完整小組審過),**無巨集 `.pptx`**(見「安全:禁止巨集」)。
- `review-report.md` —— 逐章記錄完整小組各審查者第幾輪通過、用了哪些例子;以及素材篩選結果摘要。這份報告就是驗收證據。

### 90_Excluded — 排除的資料 + 封存的舊版產出

這個資料夾有兩種用途,在 `90_Excluded/EXCLUDED.md` 裡**分開兩區**記錄,別混在一起:

- **排除的素材**:評估後**不適合用**的素材移來這裡(不要刪除,保留讓使用者覆核),逐筆記一行原因(例如:離題、重複、解析度太低看不清、過期版本)。
- **封存的舊版產出**:重跑時把 `30_Output/` 既有的舊版簡報加時間戳移來這裡保存(見第 8 步)。這些是過去版本、不是不適合的素材,要標明清楚。

## 開工前必問(最多三個一定要問的設定)

在開始產出投影片**之前**,以下事項一定要問使用者,不要自己預設:

### 1. 母片(若 `00_Masters/` 有檔案)

- **空的 / 不存在**:不套母片,從零做(走 `pptx` skill 的 `pptxgenjs.md`)。
- **剛好一個檔**:問「要套用這個母片嗎?(檔名:`<x>`)」。要 → 套用(走 `pptx` skill 的 `editing.md`);不要 → 從零做。
- **超過一個檔**:列出所有檔名,問「要套用哪一個母片?」(並提供「都不套」)。選定後才往下。

套母片只決定外觀,不影響「素材篩選 + 每章受眾審查 + 3 個例子」這些品質關卡。

> **一律用「投影片母片」(Slide Master),不要用「備忘稿母片」或「講義母片」。** PowerPoint 有三種母片,作用不同:**投影片母片(Slide Master)+ 版面配置(Slide Layouts)** 決定投影片本身的外觀與字級,這才是要套的;**備忘稿母片(Notes Master)** 只管「備忘稿列印頁」的樣子,**講義母片(Handout Master)** 只管講義列印——這兩個都**不可**拿來決定投影片外觀 / 字級。在 `.pptx` 結構裡,要參照的是 `ppt/slideMasters/` 與 `ppt/slideLayouts/`,**不是** `ppt/notesMasters/`、`ppt/handoutMasters/`。下一節「字級完全照母片」指的字級,也一律以**投影片母片 / 版面配置的佔位字級**為準。

### 2. 長寬比

一定要問使用者要用哪種長寬比,選項至少包含:

- **16:9 寬螢幕**(13.333 × 7.5 吋)—— 現代投影 / 螢幕最常用。
- **4:3 標準**(10 × 7.5 吋)—— 舊投影機 / 部分企業範本。
- **其他 PowerPoint 支援的比例**:例如 16:10、A4、Letter、橫式 / 直式,或使用者自訂的吋 / 公分尺寸。

若使用者**同時選了母片**,而母片本身的尺寸與所選長寬比不同,要回頭確認「以母片尺寸為準,還是覆蓋成你選的長寬比?」——預設尊重母片,除非使用者明確要覆蓋。

### 3. 沿用 / 延伸上一版(僅當 `30_Output/` 已有簡報時才問)

**重跑時**,如果 `30_Output/` 已經有上一版簡報,**一定要先問**使用者要走哪一種——因為使用者可能已經**手動修改過**那份簡報,那些修改不該被無聲蓋掉:

- **(A) 以上一版為基礎延伸**:把 `30_Output/` 既有的簡報當成基礎(它可能含使用者的手動修改),只在其上套用這次要的變更——走 `pptx` skill 的 `editing.md` 編輯既有檔,**保留既有內容與手動編輯**。此時上一版簡報等同「參考底稿」。
- **(B) 從 `20_References/` 重新產生**:不沿用手動修改,照標準流程從參考資料重建。

無論選哪一種,動手前都**先把既有簡報加時間戳封存到 `90_Excluded/`**(見第 8 步),手動修改的版本不會遺失。若使用者選 (A) 但 `20_References/` 也有新素材,延伸時一併納入。

## 安全:禁止巨集

為安全起見,**產出的簡報禁止啟用巨集**:

- 輸出一律存成 `.pptx`,**絕不**輸出 `.pptm` / `.potm` 等啟用巨集的格式。
- 即使 `00_Masters/` 的母片是 `.pptm` / `.potm` 或內含 VBA / 巨集,**都不可把巨集帶進產出** —— 套用其外觀,但另存為乾淨的 `.pptx` 並去除任何 VBA / 巨集(打包後不應有 `ppt/vbaProject.bin`)。
- 若母片明顯依賴巨集才能運作,告知使用者該母片含巨集、產出會以無巨集版本呈現。

## 工作流程

整體順序:**檢查 OUTLINE.md / SKILL 版本是否變更 → 整理素材(暫存→採用/排除)→ 消化內容(OUTLINE.md+補充素材,不足可上網查)→ 定審查小組(完整小組)→ 開工前必問(母片 + 長寬比 + 若有舊版問沿用/重產)→ 排章節 → 逐章寫(含 3+ 例子 + 三方視角欄位)→ 逐章審查(完整小組全員)→ 修到全員通過 → 產 pptx(封面 + 目錄 + 無巨集 + 進度條 + 來源網址)→ QA → 更新 INDEX.md / README.md / CLAUDE.md(含 OUTLINE 指紋 + SKILL 版本)**。素材篩選與審查是兩道擋在產出前的關卡。

### 0. 檢查 OUTLINE.md 與 SKILL 版本是否變更(每次執行都做)

`OUTLINE.md` 是內容主來源,所以每次跑**先比對它有沒有變**:

- 算 `OUTLINE.md` 目前的指紋(對內容做雜湊,例如 sha256 取前 12 碼),跟**上次記在 `INDEX.md` 的指紋**比;若 `30_Output/OUTLINE.used.md`(上次產出所用的快照)存在,也可直接 diff 看**哪幾章變了**。
- **首次 / 沒有舊指紋**:當作新專案,照常往下。
- **有變更**:告訴使用者變了(有快照就指出哪些章節新增 / 刪除 / 修改);這些章節要重做。這也接到「開工前必問」的沿用/重產判斷。
- **未變更**:告訴使用者 `OUTLINE.md` 自上次產出後沒變。若 `30_Output/` 已有簡報、且 `20_References/`、`10_Input/` 也沒新素材,就提醒使用者「內容來源沒變,確定要重跑嗎?」讓他決定(可能只是想換母片或長寬比)。

**同時比對 SKILL 版本**:讀本 skill frontmatter 的 `version`(日期式,如 `2026-06-28`),跟**上次記在 `INDEX.md` 的 SKILL 版本**比:
- 不同(或 INDEX 沒記過)→ 告訴使用者「**ref2pptx 規則自上次產出後更新過(舊版 → 新版)**,這份簡報是用舊版規則做的,建議重跑以套用新規則」。
- 相同 → 不用特別說。
- 維護慣例:**每次改動 SKILL.md,就把 `version` 改成「當天的實際日期」**(格式 `YYYY-MM-DD`,例如今天是 2026-06-28 就寫 `2026-06-28`)。**重點:日期要跟著真實日期走,不要沿用上次的舊日期**;同一天若改多次、想區分,再在後面加小數(`.2`、`.3`)。這個欄位一定讀得到(在 frontmatter、隨 skill 載入),所以跨環境都能比對,不必去磁碟找檔。

**順便檢查 skill 版本有沒有更新**:讀自己 frontmatter 的 `version`,跟**上次記在 `INDEX.md` 的 skill 版本**比:
- 不同(或更新)→ 告訴使用者「ref2pptx 自上次產出後有更新(舊版 `<x>` → 新版 `<y>`)」,並建議**重新產生**以套用新規則。
- 相同 / 首次 → 不必特別說。
- (進階 / 選用:若能定位到 skill 安裝路徑的 `SKILL.md`,可改用對該檔做 sha256 來自動偵測任何改動;定位不到就以 `version` 欄位為準。)
收尾(第 10 步)時把目前 `version` 一併寫進 `INDEX.md`。

### 1. 整理素材(暫存 → 採用 / 排除)

把素材篩成乾淨的一組再開始做簡報:

- **把 `10_Input/` 的每一件都當成「第一次看到」**:使用者可能把先前排除、現在想重評的東西**手動從 `90_Excluded/` 移回 `10_Input/``**——因為簡報內容會不斷更新,以前不適用的,現在可能用得上了。所以一律依**現在的主題與簡報狀態**重新判斷,**不要**因為它「以前被排除過」就再排除,也**不要**去翻 `90_Excluded/EXCLUDED.md` 的舊判斷來預設立場。判斷相關性時對照的是當下的內容(若這次是「延伸」上一版,就連同既有簡報一起對照)。
- **OUTLINE 有新增章節時,才主動回頭挖 `90_Excluded/`**:若第 0 步偵測到 `OUTLINE.md` **新增了章節**,就掃一遍 `90_Excluded/` 裡「排除的素材」,對照新章節與主題,看有沒有**之前棄用、這次用得上**的(例如那時離題、現在新章節剛好需要)。判斷「當第一次看到」。(沒有新增章節就不必每次掃。使用者仍可隨時手動把東西移回 `10_Input/`。)
  - 現在可用 → 移回 `20_References/`(資料夾比照規則補 `NN_` 索引),並從 `EXCLUDED.md` 的「排除的素材」區移除該筆。
  - 仍不適用 → 留在原處,不必動。
  - **只看「排除的素材」,不要動「封存的舊版產出」**(那些是歷史簡報,不是素材)。
  - 在 `review-report.md` 記一筆:這次從 `90_Excluded/` 復用了哪些、為什麼現在可用。
- 逐件看 `10_Input/`(必篩)與 `20_References/` 裡的素材。讀檔前先用 `file-reading` skill 判斷怎麼讀。
- **`10_Input/` 裡若是資料夾,整個資料夾當一個單位處理**:把它當成一組一起評估,然後**整個資料夾**移到 `20_References/`(採用)或 `90_Excluded/`(排除),不要拆開裡面的個別檔案分開搬。
- **清掉 `20_References/` 裡用不到的素材**:做完內容消化 / 產出後,`20_References/` 裡若有**這次完全沒被任何章節採用**的素材,就把它**移到 `90_Excluded/`**(用移動、不要刪除),並在 `EXCLUDED.md` 的「排除的素材」記一筆原因(例如「本次未採用」)。**若該檔在某個資料夾內,整個資料夾一起移**(跟採用時「資料夾整包移動」一致),不要只挑單檔搬。之後若又用得上,會在 OUTLINE 新增章節時、或該資料夾被擴充時從 `90_Excluded/` 撈回(見上 / 見下)。這讓 `20_References/` 只留「這份簡報實際有用到」的素材、保持乾淨。
- **檢查 `20_References/` 與 `90_Excluded/` 裡資料夾的內容有沒有變(可能後來又擴充了資料)**:這兩個區的資料夾不是一成不變——使用者可能事後往某個既有資料夾**再丟新檔案 / 擴充內容**。所以每次執行都比對各資料夾**目前的內容**(檔案清單 + 大小 / 修改時間,或內容指紋)跟**上次記在 `INDEX.md` 的**有沒有不同:
  - `20_References/` 裡某資料夾**多了 / 改了檔案** → 那組素材被擴充了,要**重讀並把新內容納入**對應章節(不是只沿用上次讀過的)。
  - `90_Excluded/` 裡某「排除的素材」資料夾**多了 / 改了檔案** → 它可能因為補了資料而**變得可用了**,要**重新評估**;可用就比照規則移回 `20_References/`。(「封存的舊版產出」不在此列,不重評。)
  - 在 `INDEX.md` 記下這兩區各資料夾目前的內容清單 / 指紋,供下次比對。
- **判斷適不適用**:跟主題相關嗎?品質 / 可讀性夠嗎(圖太糊、字太小看不清?)?跟別的素材重複嗎?是不是過期版本?
- **採用**:適用的素材留在 / 移到 `20_References/`。
  - **採用的若是資料夾,移進 `20_References/` 時要在原資料夾名稱前加數字索引前綴**(格式 `NN_原資料夾名`,兩位數補零、底線分隔,例如 `白板筆記` → `03_白板筆記`),這樣 `20_References/` 內的排序才不會亂掉。
  - **索引怎麼取**:看 `20_References/` 現有帶 `NN_` 前綴的項目(檔案與資料夾一起算),取最大號 +1;沒有就從 `01` 起。一次採用多個資料夾就依序給連號(`03`、`04`…)。已經有 `NN_` 前綴的資料夾不要重複加。
- **排除**:不適用的**移到 `90_Excluded/`**(用 `mv`,**不要刪除**);若是資料夾就整個移過去。並在 `90_Excluded/EXCLUDED.md` 加一行寫清楚名稱與排除原因(排除區不需要加索引前綴)。`EXCLUDED.md` 以「現在 `90_Excluded/` 裡實際有什麼」為準:同一項再次被排除就更新該行(不要堆疊重複),已被使用者移回 `10_Input/` 而不在 `90_Excluded/` 的舊項可清掉。
- `20_References/` 是使用者刻意放的「要用」的素材:除非明顯無法使用(壞檔、完全離題),否則優先採用;若要把 `20_References/` 裡的東西排除,**先告知使用者**再移。
- **`(O)` 優先記號**:名稱裡含 `(O)` 的檔案或資料夾,是使用者標記的**優先素材**——一律採用,並標為高優先來源;之後寫章節、選素材、取例子時**優先使用**這些。搬進 `20_References/` 時保留 `(O)`(若同時要加索引前綴,放在前綴之後,例如 `03_白板筆記(O)`)。
- **`URL.txt` 來源網址**:採用的**資料夾裡若有 `URL.txt`**,代表那是這份素材的**來源網址**。**檔名不分大小寫**:`url.txt`、`URL.txt`、`Url.txt`、`URL.TXT` 等都算數(比對時一律轉小寫再比)。讀出裡面的網址,記住「這個資料夾 → 這個網址」的對應;用到這份素材內容的投影片,要在頁面標上來源(見第 8 步進度條設計)。`URL.txt` 本身是中繼資料,不當成簡報內容、也不列入排除。
- 篩完把結果摘要(採用幾件 / 幾個資料夾、各自新索引、`(O)` 優先項、有來源網址的資料夾、排除幾件與原因)記進 `30_Output/review-report.md`。

### 2. 消化內容(主來源 OUTLINE.md + 補充素材)

先讀**根目錄 `OUTLINE.md`**——它是投影片的章節結構與各章內容的主來源。再讀 `20_References/` 的**補充素材**(圖、引用文件、數據來源),整理出每章可用的真實案例 / 數據 / 圖。標記哪些概念**天生帶技術門檻**(之後審查要特別盯)。內容衝突時**以 `OUTLINE.md` 為準**,`20_References/` 用來豐富 / 佐證;補充素材若有 `(O)` 優先記號,在補充取捨時優先採用。同時建立「哪段內容用到哪個素材、那個素材 / 資料夾有沒有 `URL.txt` 來源」的對應,供產出時標來源用。

**素材不足時可上網查(要留網址)**:若 `OUTLINE.md` + `20_References/` 不足以支撐某章(缺定義、缺實例、需要較新的事實或數據),**可以上網搜尋來補**。但有兩個硬規定:
- **凡是用到網路查來的內容,該頁一定要留下來源網址**(做法同 `URL.txt`,標在進度條上方,見第 8 步)。沒留網址就不能用網路內容。
- 用可信來源(官方文件、原始出處優先),**不要照抄整段**——用自己的話改寫、只取重點;不確定就標示存疑或回報使用者,不要編造。
- 仍以使用者自己的素材優先,網路只是補洞;補了什麼、用了哪些網址,記進 `review-report.md` 與 `INDEX.md`。

### 3. 確定受眾人物誌

先讀**專案根目錄的 `AUDIENCE.md`**(不在 `20_References/`)。沒有就用下方預設庫,並向使用者確認。每位受眾至少要寫清楚三件事,審查時才有判準:

- **背景**:他會什麼、不會什麼(這是審查的核心)
- **日常工作**:他每天在處理什麼,例子要貼著這個給
- **看不懂就會卡住的東西**:哪類術語 / 假設前提對他是黑箱

**聽眾學歷基準(全體適用,凌駕個別人物誌的『不懂』判斷)**:假設**所有聽眾都大學畢業**——一般知識、閱讀理解、邏輯推理都是大學程度。由此:
- **不要過度簡化 / 太一般化**:不用幼稚的比喻、不把常識當新知從頭解釋、不把聽眾當外行小白。寫給「受過高等教育、只是不熟這個領域」的成年人看。
- **技術字眼以「高中就學過的」為基準**:高中(含)以下程度的科學 / 數學 / 資訊名詞(例如機率、函數、變數、流程圖、資料庫概念這類)**可直接用、不必解釋**;**只有超過高中程度的專業術語才需要解釋**(或用「縮寫角落註」帶出全名)。
- 各人物誌(PM / QA / RD / 終端用戶)列的「不會 / 看不懂」是**在這條底線之上**再加各自的專業限制——例如 PM 不懂程式架構、QA 不會寫 code,但他們都看得懂高中程度的名詞、不需要被當小學生講解。

預設受眾庫(可直接用、可改、可加):

| 代號 | 受眾 | 會 / 不會 | 例子要貼近 |
|------|------|-----------|-----------|
| `PM-nontech` | 完全沒有技術背景的 PM | 懂產品與流程,不懂程式 / 架構 / 指令;可接受英文技術縮寫(每張投影片角落都附全名) | 需求、排期、溝通、驗收 |
| `PM-tech` | 技術轉型 PM(原本是 RD 或 QA) | 有技術底、看得懂術語與技術內容;工作是把外界來的資訊消化後,用**通識語言**講給其他人聽、或跟**客戶討論規格**——很在意「這段能不能轉述成讓非技術的人 / 客戶也聽得懂」 | 把技術內容翻成白話轉述、跟客戶談規格 / 釐清需求、技術 ↔ 商業之間的橋接 |
| `QA-nocode` | 不會寫 code 的 QA | 懂測試設計與情境,不會自己寫程式 | 測試案例、缺陷重現、驗收條件 |
| `QA-nolinux` | 不熟 Linux 的 QA | 會測試,但對指令列 / 伺服器環境陌生 | 用 GUI / 圖形工具完成同一件事 |
| `QA-guionly` | 只會測 GUI、無法揣摩使用者的 QA | 會照案例點 GUI 驗收,不碰底層;難以自行推想使用者意圖 | 「使用者想做什麼→怎麼點→看到什麼」寫到具體,情境寫明不靠腦補 |
| `RD-spec2code` | 只會把 spec 翻成 code 的 RD | 能照規格實作,較少做架構 / 取捨判斷 | 規格如何對應到實作、邊界條件 |
| `USER-zen` | 佛系終端用戶 | 什麼都不懂、只負責用;遇到 bug 自然繞過,不發問、不抱怨 | 直接照做就會的步驟;不能假設他會主動動腦 |
| `USER-picky` | 找碴終端用戶 | 什麼都不懂;看不懂 / 不會用就直接說「東西爛」 | 清楚到挑不出毛病、零含糊零跳步的內容 |

**審查小組(完整小組,全員都要過)**:每章都由**完整小組**審查,**全員通過才算過**。成員 = **PM / QA / RD + USER-zen + USER-picky**。各成員用哪個人物誌在 `AUDIENCE.md` 指定,預設:PM = `PM-nontech`、QA = `QA-nocode`、RD = `RD-spec2code`、USER-zen(佛系)、USER-picky(找碴)。在 `30_Output/review-report.md` 記下成員與通過情形。

各成員的通過條件:
- **PM / QA / RD**:對該角色沒有未解釋的術語 / 黑箱假設;找得到至少一個貼近他、看得懂的例子;能從頭跟到尾不卡。
- **`USER-zen`(佛系)**:他不會發問,看不懂就默默掉隊;通過 = 被動讀也能跟到底,沒有需要自行推敲的斷點。
- **`USER-picky`(找碴)**:看不懂就一直抱怨;通過 = 找不到明顯可被他挑成「看不懂 / 不會用」的地方。

### 4. 排出章節大綱

**章節結構與順序一律以根目錄 `OUTLINE.md` 的標題順序為準**,不要自行重排或增刪章節。`OUTLINE.md` 各章標題下的內容就是該章主體;`20_References/` 的補充素材歸到對應章節。每章之後都交完整小組審查。(若使用者沒有 `OUTLINE.md`,才依素材自行擬章節並先跟使用者確認。)

### 5. 逐章撰寫(內含 3+ 例子規則)

每章撰寫時就**內建**驗收標準 3.B:

> 每個章節必須有**三個以上**例子,且要**涵蓋 PM / QA / RD 三方視角**——三方各自都要找得到至少一個對他有意義的例子(三個剛好可一方一個,需要時更多)。

例子要具體到該角色能對號入座(用他工作中真的會碰到的情境、名詞、畫面),不能是抽象換句話說。例子盡量涵蓋不同面向(例如:正常情況 / 例外情況 / 對比錯誤做法),而不是同一個例子講三次。**寫例子時就避開法律風險**:不用刻板印象、不點名真實同事 / 主管 / 競品當負面例子、不放性暗示或歧視 / 霸凌字眼(詳見第 9 步「法律風險 QA」)。

**「三方視角」是每個主題的必備欄位**:每個主題都要有一個「三方視角」區塊,底下三行,分別用 `[PM]`、`[QA]`、`[RD]` 標籤開頭,各寫**該角色對這個主題的具體角度 / 例子**(一行一個角色,具體、不抽象)。再加一句 **「簡單來說:…」** 的白話總結收尾(同一風格,建議比照)。這三行剛好就是 3.B 要求的三方例子的落地呈現。**若一個主題跨多頁,三方視角只放在該主題最後一頁、續頁標題加 (cont'd)**(見第 8 步)。渲染規格見第 8 步「三方視角欄位」。

**英文技術縮寫規則**:縮寫**不在內文展開全名**(保持內文乾淨);改成**每一張用到該縮寫的投影片,都在角落用極小字列出「縮寫 = 全名」**(例如 `API = Application Programming Interface`)。同一張頁出現多個縮寫就列多行;同一縮寫每次出現的那幾頁都要附(不是只附一次)。渲染規格見第 8 步「縮寫全名角落註」。因為每頁都查得到全名,PM 可接受內文直接用縮寫。

### 6. 逐章審查(完整小組,全員通過)

每寫完一章,用完整小組各成員的視角各審一次。建議用多個 subagent 各帶一個角色設定獨立審查 —— 因為你剛寫完,會「看到你以為有寫清楚的」,而不是「實際在頁面上的」,換別人的眼睛才抓得到盲點。

審查 prompt 範本:

```
你是 <審查者:人物誌代號:背景一句話>。
你會的:<...>。你不會、看了會卡住的:<...>。
請只用這個身分讀以下章節,並回答:
1. 有沒有出現你看不懂、卻沒被解釋的術語或假設?逐個列出。
2. 這章有沒有至少一個貼近你工作、你看得懂的例子?
3. 哪幾頁你會跟不上、需要重看?

<章節內容>
```

**通過條件**:完整小組(PM / QA / RD + USER-zen + USER-picky)**全員都過才算過**。USER-zen、USER-picky 通常會逼出「要更白話、把情境講更死、補更多直觀例子」。各方有衝突時(例如 RD 想要更多技術細節、終端用戶想更白話),以「**全員都能跟上**」為準:白話講主線、技術細節放補充或備忘。

### 7. 修到通過

針對審查抓到的問題改章節(補白話、換更貼近的例子、拆頁),然後**重審受影響的部分**。每章在 `30_Output/review-report.md` 記錄:各審查者第幾輪通過、用了哪些例子。修到全章節**完整小組全員通過**。

### 8. 產出 pptx(無巨集 + 進度條)

所有章節通過後,才把內容做成投影片,**輸出到 `30_Output/`**。產出時套用前面「開工前必問」拿到的設定:

- **先封存舊版(不覆蓋)**:產新版**之前**,先檢查 `30_Output/` 有沒有同名的既有簡報;有的話,把它**加時間戳後移到 `90_Excluded/`**(例如 `<簡報名>_YYYYMMDD-HHMMSS.pptx`),並在 `90_Excluded/EXCLUDED.md` 的「封存的舊版產出」區記一筆。這樣每次重跑都會留下舊版、不會被覆蓋(含使用者手動修改的版本)。用移動不要用刪除。(只封存簡報;`review-report.md` 會在這次重跑時一併更新。)
- **依「沿用 / 延伸」選擇決定怎麼做**(見開工前必問第 3 點):
  - **延伸 (A)**:以剛封存的那份(即使用者可能手動改過的上一版)為**基礎底稿**,走 `pptx` skill 的 `editing.md` 在其上編輯,**保留既有頁面與手動修改**,只套用這次的變更;若 `20_References/` 有新素材,一併納入。新增 / 改動的章節仍要過受眾審查與 3 例規則。
  - **重新產生 (B)**:從 `20_References/` 重建(下列其餘規則照走)。

- **產出一份簡報**:做出一份 `<簡報名>.pptx`(完整小組審過的版本),套用下列設定(母片、長寬比、無巨集、進度條、來源網址、三方視角欄位)。封存、`OUTLINE.used.md` 快照、QA 都對這份做。
- **目錄頁(封面之後放一頁)**:封面之後、進入內容之前,固定放一頁**「目錄」**:
  - 標題「目錄」(或「課程大綱 / Agenda」),其下**依 `OUTLINE.md` 的章節順序列出每一章標題**(建議加編號 1、2、3…);章名要跟內文各章、底部進度條的章節一致。
  - 這是**結構頁**,比照封面 / 章節分隔頁:**不放進度條、來源網址、三方視角面板**。
  - 樣式套母片;章很多排不下就分兩頁(第 2 頁標題加 (cont'd))。
  - 章節有增刪時,目錄要跟著更新(以 `OUTLINE.md` 為準)。
- **版面安全:不出界、不蓋標題(硬規則)**:
  - **所有形狀都必須完整落在投影片畫面內**(座標範圍 0..寬 × 0..高;16:9 = 0..13.333 × 0..7.5 吋)。**絕對不能把任何東西放到畫面外**(例如座標算錯把進度條標籤丟到 y≈183 吋)。畫面外的形狀會變成「看不見卻存在」的物件,撐長編輯檢視的捲動範圍,**害滑鼠滾輪滾不到下一頁**——附上傳的 `example.pptx` 有 238 個這種出界形狀,就是錯誤示範。
  - **不留任何殘留 / 垃圾形狀**:每頁只放真正要顯示的東西;算完座標要驗證在界內,別讓暫存、算錯、或被取代的形狀留在畫面外或重疊處。
  - **內文絕對不可蓋到標題(最常見的錯,務必做到)**:內文 / 內容文字框的**上緣(y)必須落在標題下緣之下**,中間留一點間距,兩者不可重疊。做法:
    - **先讀出標題的下緣**:標題佔位框的位置來自版面 / 母片(本範例母片 `title` 在 y≈0.20..0.67),所以標題下緣 ≈ 0.67 吋。
    - **內文框的 top 設在標題下緣之下**(例如 ≥ 0.8 吋,留約 0.1 吋間距);**最簡單可靠的做法是直接沿用母片的內文佔位框**(本範例母片 `body` 在 y≈0.95..6.93,本來就在標題下方)。
    - **絕對不要把內文框的 y 設成 0 / 設成投影片最頂**——這是目前一直發生的錯誤(內文框被放到 y=0,整個壓在 y=0.2 的標題上)。
    - 內文太長就靠 autofit 縮、或拆多頁,**不要靠把內文往上移到標題區來騰空間**。
  - 同理,三方視角面板不蓋內文、進度條不蓋頁碼(見各自規則)。**總原則:任兩個元素不重疊、且全部在畫面內。**
- **母片**:選了母片就用該母片做(走 `pptx` skill 的 `editing.md`);沒選或無母片就從零做(`pptxgenjs.md`)。
- **字級照投影片母片為基準 + 啟用 autofit**:這裡的「母片」一律指**投影片母片(Slide Master)/ 版面配置(Slide Layouts)**,**不是備忘稿母片**。各文字層級(標題、內文、項目等)**以投影片母片定義的字級為基準,不要手動改成別的數字**——**字偏小是母片本身的設定,不另外補償放大 / 縮小**。同時**在文字框啟用「文字溢出時自動縮小(Shrink text on overflow / autofit)」**。內容(含三方視角、簡單來說)套到對應層級就沿用該層級基準字級。例外:本 skill 自訂、疊在母片之外的極小附加註記(進度條標籤、來源網址、縮寫角落註)是獨立小字,不在此限。
- **⚠ autofit 的致命陷阱(務必懂)**:用程式產出時若只寫一個**空的 `<a:normAutofit/>`(沒有 `fontScale`)**,**PowerPoint 不會自己縮**——它照檔案裡存的縮放(等於 100%)算,文字就用母片原始大小**溢出框、蓋到下面的三方視角面板**(這正是 example.pptx 那四頁的成因)。但 **LibreOffice 會即時重算縮放**,所以「轉圖看起來正常」會騙過你。要避免:
  - **首選:讓內容在母片字級下本來就放得進框**(內文段數控制在框放得下的量,超過就**拆頁**——見「多頁主題」)。這樣根本不必靠 autofit,PowerPoint / LibreOffice 都不會爆。
  - **若要靠 autofit 縮**:就得在 `normAutofit` 裡**實際算好並寫入 `fontScale`(必要時加 `lnSpcReduction`)**,例如 `<a:normAutofit fontScale="85000" lnSpcReduction="10000"/>`,讓 PowerPoint 照這個縮放顯示;**不可只丟空的 `<a:normAutofit/>` 就當作有縮**。
  - 因為 LibreOffice 轉圖會自動縮、掩蓋這個問題,**視覺 QA 不能只信轉出的圖**:要同時用「段數 / 行數 vs 框高」估算在母片字級下是否塞得下,塞不下就拆頁或寫入 fontScale。
- **內文有「編號標記」就用自動編號清單呈現**:當素材 / `OUTLINE.md` 的內文是**用編號列出的**(像 `1. ` `2. ` `3. `、`1) `、`(1) `、`一、二、` 等),投影片要用 **PowerPoint 的「自動編號清單」**呈現(段落套 `buAutoNum`,阿拉伯數字加句點用 `type="arabicPeriod"`),**而不是把號碼當普通文字打進去**。做法:
  - **把手動打的號碼從文字裡拿掉**(例如 `1. GenAI 時代…` → 文字只留 `GenAI 時代…`),改由清單格式自動產生號碼;否則會「手動 1.」加「自動 1.」**重複編號**。
  - 對照範例:`example.pptx` 是把 `1. 2. 3.` 打進文字(錯誤示範);`example_Modified.pptx` 是文字不含號碼、段落套 `buAutoNum:arabicPeriod`(正確)。
  - 自動編號的好處:增刪項目會**自動重新編號**、縮排層級正確、樣式統一。
  - 若內文是**無序要點**(沒有特定順序),則用**項目符號清單**(`buChar`)而非編號;有先後 / 步驟 / 條列順序的才用編號。
- **長寬比**:用使用者選定的比例設定投影片尺寸(16:9 = 13.333×7.5 吋、4:3 = 10×7.5 吋,或使用者指定的其他尺寸)。
- **禁止巨集**:一律存成乾淨的 `.pptx`,不帶任何 VBA / 巨集(見「安全:禁止巨集」)。
- **多頁主題(分頁 + cont'd)— 裝不下就「強制」拆頁,不要硬塞**:一個主題(topic)一頁放不下時,**必須**拆成多頁,不可把內文硬塞到爆框、也不可靠 autofit 縮到看不清。判斷標準是「**內文要能完整顯示在內文框內、且下方還留得出三方視角面板(含與內文約 0.5 吋間距)**」:
  - **內文量的上限**:內文框高度有限(本範例母片內文區約 0.95..4.65 吋、約 3.7 吋),在母片可讀字級下大概只放得下**少數幾行 / 約 6–10 段**。**超過就拆頁**——不要塞到 10 段、20 段、甚至 45 段(附上傳的 `example_2.pptx` 就是一頁硬塞約 45 段,面板再怎麼移都會被爆出來的內文蓋住,這種一定要拆)。
  - **拆法**:把內文分到多頁,**第 2 頁起標題後面加「(cont'd)」**;每頁內文都要落在內文框內、不爆框。
  - **三方視角只放最後一頁**:同一主題的多頁中,「三方視角」面板**只放在最後一頁**(且那頁的內文也要短到留得下面板);前面的續頁不放面板、專心放內容。
  - 其餘每頁元素(進度條、來源網址、縮寫角落註)仍**每頁照放**。三方審查針對整個主題(看完所有頁)進行,並在最後一頁呈現三方視角。
  - 寧可**多一頁、每頁清爽**,也不要一頁塞爆讓面板蓋住內文。
  - **拆頁的技術注意(在既有 pptx 上「加頁」時,走 `editing` 路線)**:用程式新增投影片,必須同步更新四處——① 新的 `ppt/slides/slideN.xml` 及其 `_rels/slideN.xml.rels`(至少連到 `slideLayout`);② `[Content_Types].xml` 加該 slide 的 `Override`;③ `ppt/_rels/presentation.xml.rels` 加一條 slide 關係;④ `ppt/presentation.xml` 的 `<p:sldIdLst>` 加 `<p:sldId>`(用沒用過的 id)並排到正確順序(順序由 sldIdLst 決定,不是檔名)。**最關鍵的雷:新關係的 `rId` 不可與 `presentation.xml.rels` 既有的撞號**——既有 rId 常被 `slideMaster / notesMaster / presProps / viewProps / theme / tableStyles / customXml` 佔走(例如有的檔 `rId10–rId14` 已被佔),**先掃出目前最大的 rId、從 +1 開始發**;撞號會把 theme / notesMaster 等覆蓋掉,**整份簡報就壞了**(徵兆:投影片變成顯示母片佔位提示字、或跑出別頁的備忘稿內容)。若是**整份重產**(`pptxgenjs` 一次建好所有頁)則沒有這個撞號問題,首選重產。
- **三方視角欄位(每個主題必備;多頁主題放在最後一頁)**:主內容下方,固定放一個「三方視角」區塊。**呈現樣式(沿用使用者偏好的版本)**:
  - **底層放一個淺色圓角面板**(圓角矩形 roundRect),把標題 + 三行 + 簡單來說都框在裡面;面板填很淡的底色(預設淺藍灰 `EDF2F7`,或改用母片配色的淺色調),無邊框或極淡邊框。
  - **小標題「三方視角」**:用**第 5 個顏色**——必須**和下面四種顏色(PM 藍、QA 綠、RD 紫、簡單來說 灰)都不同**(預設用深橙 `C2410C`,或母片色系中一個與那四色明顯區隔的強調色);**粗體**;**字級比下面三行大 4 pt**(例如三行 10pt 時,標題 14pt)。
  - 其下**三行,每行約 10pt,各以 `[PM]` / `[QA]` / `[RD]` 開頭,整行用該角色的專屬色**:`[PM]` → 藍 `005B96`、`[QA]` → 綠 `007A5E`、`[RD]` → 紫 `6B4C9A`(沒有母片特定配色時用這組;有的話可換成母片色系裡三個夠區隔的顏色)。每行寫該角色對本主題的具體角度 / 例子,一行一個角色。
  - 最後一行 **「簡單來說:…」**:灰色 `555555`、**斜體**,白話總結。
  - **面板內所有文字(含「三方視角」標題、三行、簡單來說)一律靠左對齊**,標題不要置中。
  - 三個標籤與順序固定(PM→QA→RD),跨頁一致;這是必備欄位,缺了視同未完成。
  - **位置 — 絕對不能蓋住內文,且要留「實際間距」**:面板是**獨立色塊**,會遮住底下的東西,所以**上緣必須在內文文字結束之後**,而且要留**實際看得出來的間距(不能只差一點點)**——附上傳的 `example.pptx` 面板上緣 4.75 吋、內文框底 4.65 吋只差 0.1 吋,內文一溢出就被蓋到,這是錯誤示範;修正版把面板**往下移到上緣 ≈ 5.43 吋**(與內文約留 0.7–0.8 吋),就不蓋了。做法:把面板放到下半部、與內文之間留約 **0.5 吋以上**的空白;**注意「蓋到」看的是內文『實際文字』有沒有溢出框、撞到面板,不是只看內文框的標示高度**——內文太多、autofit 縮了還是爆框時,**不要只把面板往下挪(挪到最後還是會被爆出來的內文蓋住,如 `example_2.pptx`),而是要減量或拆成多頁**(見「多頁主題」)。
  - **下緣**只要留在「來源網址 / 進度條」之上即可,可以**比以前更靠下**(修正版面板底緣到 ≈ 6.73 吋、仍在進度條 6.85 之上);亦即面板被夾在「內文之下(留實際間距)、來源網址/進度條之上」的帶狀區,**優先確保不蓋內文**。
  - 封面 / 章節分隔頁可省略。
- **進度條 / 小大綱(每頁底部)**:每張**內容頁**底部都要放一條進度指示,讓受眾隨時知道「現在到哪、還剩多少」。做法見下方「進度條設計」。封面 / 章節分隔頁可省略。
- **縮寫全名角落註**:每張用到英文技術縮寫的投影片,在**角落**用極小字列出「縮寫 = 全名」。要求:
  - **字級比簡報內文最小字級再略小一級**(會是全頁最小的字;若內文最小是 N pt,這裡用比 N 再小一點、仍勉強可辨識的尺寸,例如同來源網址那一級或更小)。
  - **不影響排版**:用絕對定位浮在角落,不擠壓 / 不推動內文。
  - 放在**不與進度條、來源網址、頁尾、頁碼衝突**的角落(例如右下或左下角的剩餘空間);跨頁位置一致。
  - 一頁多個縮寫就列多行(極小字);長寬比改變時座標跟著換算。

#### 進度條設計

把它做成**有意義的小大綱**,不是裝飾色條(`pptx` skill 禁止橫貫整頁的裝飾條,但帶資訊的進度 / 導覽元件是合理例外,前提是要做得像導覽、不像填充)。

- **形式**:沿頁底排成 N 個分段(N = 章節數),一段對一章,段與段之間留小間隙(分段式,而非單一實心條)。
- **狀態用顏色區分**:已上過的章節用「完成色」、目前章節用「主色高亮」、還沒上的用淺灰。
- **加文字定位**:每段下方放極小的章節短標題(這就是「小大綱」),目前章節的標題加深;另可在角落放「2 / N」的頁章標示。
- **位置**:靠頁底,但**不可蓋到頁碼**——往下移、貼近底緣(例如 16:9 在 y ≈ 6.85 吋),並把**頁碼擺在進度條沒占用的角落**(例如最底的左下或右下角,與進度條錯開)。高度約 0.12–0.18 吋,左右對齊內容邊界(約 0.6 吋邊距)。
- **一致**:每頁同一位置、同一套配色,跨頁不要跳動。
- **長寬比要跟著變**:座標依實際投影片尺寸換算(4:3、自訂尺寸時不要沿用 16:9 的 y 值)。
- **章節很多時**(例如 > 8 章):分段太細,改用「目前章節高亮的一行小標題列 + 一條細進度比例」或只顯示鄰近章節。

#### 來源網址(進度條上方)

若某頁的內容來自帶 `URL.txt` 的資料夾,**或來自第 2 步的上網查資料**,在**進度條正上方**放一行**來源網址**:

- **位置**:進度條正上方一點點(16:9 約 y ≈ 6.6 吋),不要蓋到進度條或內文。
- **字很小就好**:用很小的字級(例如 7–8 pt)、低調的灰色。目的只是**讓人知道這裡有來源**,不必醒目;網址太長可截斷或只顯示網域(例如 `來源:example.com/…`)。
- 沒有任何來源(既無 `URL.txt`、也非上網查來)的頁面就不放。一頁若用到多個來源(資料夾或網路),擇主要來源放一行即可(或以 `; ` 串接,仍維持一行、極小)。
- 長寬比改變時 y 座標跟著換算,維持在進度條正上方。

> **頁尾元素由下而上、互不重疊的順序**(16:9 參考值):頁碼 / 進度條(最底,y ≈ 6.85,兩者錯開不互蓋)→ 來源網址(y ≈ 6.6)→ 三方視角面板(底緣可到 ≈ 6.7、只要在來源網址/進度條之上;上緣與內文留約 0.5 吋以上實際間距)→ 內文。其他長寬比依比例換算,維持此上下順序與不重疊。

排版、配色、字級、視覺品質一律遵循 `pptx` skill,這份 skill 不重複那些規則。把每頁的講者備忘(speaker notes)填上,方便後續實際授課。

### 9. QA

- **內容 QA**:跑 `extract-text 30_Output/<簡報名>.pptx`,檢查漏字、順序、佔位字殘留。
- **視覺 QA**:用 subagent 看轉出的圖檢查溢出 / 重疊 / 對比(照 `pptx` skill 的視覺 QA 流程)。
- **版面安全 QA(出界 / 重疊,阻擋性)**:逐頁掃描每個形狀的座標——
  - **出界檢查**:任何形狀只要有一部分落在畫面外(y 或 x < 0,或超過投影片寬 / 高)就是錯,**移回畫面內或刪除**。可程式化檢查:解開 `.pptx`,讀每張 `ppt/slides/slideN.xml` 的 `<a:off>`/`<a:ext>`,換算成吋後比對 0..13.333 / 0..7.5(其他比例照換算);發現像 y≈183 這種離譜值一定要清掉。
  - **重疊檢查**:確認**內文沒蓋到標題**(標題下緣 ≤ 內文上緣)、三方視角面板沒蓋內文、進度條沒蓋頁碼。**用轉出的圖實際看**:內文『文字』有沒有溢出框、壓到三方視角面板(座標不重疊但文字爆框也算蓋到)。**注意:轉出的圖是 LibreOffice 算的、會自動套用 autofit 縮放,可能把「PowerPoint 裡其實會爆框」的情況藏起來**;所以除了看圖,還要查每個內文框是不是「空的 `<a:normAutofit/>` 沒寫 fontScale」+ 內容量超過框——這種在 PowerPoint 會爆框。若某頁內文爆框 / 面板被內文壓住 → 該頁**內文減量或拆成多頁**(見「多頁主題」),或在 normAutofit 寫入實際 fontScale;不是只把面板挪一挪。
  - 任一頁有出界形狀、上述重疊、或內文爆框蓋到面板 = 未通過,修好再交。
- **進度條 QA**:確認每張內容頁底部都有進度指示、目前章節正確高亮、跨頁位置一致且沒蓋到內文。
- **目錄頁 QA**:確認封面之後有一頁「目錄」、且列出的章節與 `OUTLINE.md` 的章節、順序一致(沒漏章、沒多章);目錄頁是結構頁,不該有進度條 / 三方視角。
- **三方視角 QA**:確認**每個主題**都有「三方視角」面板(多頁主題在最後一頁)、且 `[PM]`/`[QA]`/`[RD]` 三行齊全(順序一致)並有「簡單來說」收尾;多頁主題的續頁標題有 `(cont'd)`。另外檢查版面:**面板沒有壓到任何內文**(上緣在內文之下)、下緣在來源網址/進度條之上;「三方視角」標題用**第 5 個(與四色不同)顏色、比三行大 4pt**;面板內文字**全部靠左對齊**。某主題缺欄位、或面板蓋到內文 = 未通過。
- **來源網址 QA**:用到帶 `URL.txt` 資料夾的頁面,確認進度條上方有那行極小來源網址、沒蓋到進度條或內文;沒來源的頁面則沒有多餘的那行。`(O)` 優先素材有確實被優先採用。
- **設定 QA**:確認長寬比與使用者選的一致;若有套母片,確認外觀有正確套上,而且套的是**投影片母片 / 版面配置**(`ppt/slideMasters/`、`ppt/slideLayouts/`),**不是備忘稿母片 / 講義母片**。
- **字級 QA**:若有套母片,確認各文字層級的**基準字級與投影片母片一致**(沒被手動改成別的數字,且來源是投影片母片、不是備忘稿母片),且文字框**已啟用 autofit(文字溢出時自動縮小)**。
- **安全 QA(禁止巨集)**:確認副檔名是 `.pptx`(非 `.pptm`/`.potm`),`unzip -l 30_Output/<簡報名>.pptx | grep -i vba` 為空(無 `ppt/vbaProject.bin`)。
- **法律風險 QA(阻擋性,不過就不能產出)**:逐頁(含標題、內文、三方視角的 `[PM]`/`[QA]`/`[RD]` 例子、講者備忘)檢查有沒有**可能讓講者被告或違反公司規範**的內容,有就**改寫或刪除**,不能留。重點掃描:
  - **性騷擾 / 性暗示**:黃色笑話、帶性意味的舉例或圖、對外貌身材的評論。
  - **歧視**:針對種族、族群、性別、性傾向、年齡、宗教、身心障礙、國籍、婚育狀態等的刻板印象、貶抑或差別待遇用語(連「舉例方便」用的刻板印象也不行)。
  - **職場霸凌 / 人身攻擊**:貶低、羞辱、嘲諷特定人或群體;用真實同事 / 主管當負面例子。
  - **誹謗 / 商譽**:對真實人物或公司做未經證實的負面指控;拿競品或他人指名道姓地貶低。
  - **隱私 / 機密**:真實個資、客戶資料、未公開的內部機密 / 營業秘密。
  - **其他法律風險**:侵權重製(整段抄他人著作 / 圖)、不實宣稱、煽動違法。
  例子是最容易出事的地方(為了「貼近」而用了刻板印象或真實人名)——特別盯。若有風險內容來自使用者提供的素材而無法安全改寫,**標示出來並回報使用者**,不要默默放進去。處理結果記進 `review-report.md`。
- **素材 QA**:確認 `90_Excluded/EXCLUDED.md` 有逐筆排除原因;簡報內容只來自 `20_References/`,沒用到被排除的素材。若這是重跑,確認舊版簡報已加時間戳封存到 `90_Excluded/`、`30_Output/` 沒被覆蓋掉舊版。
- **驗收 QA**:打開 `30_Output/review-report.md`,確認**每一章**都有完整小組全員通過標記、且列了三個以上例子(涵蓋三方)。任一章缺項 = 未通過,回第 6/7 步。
- 最後確認 `30_Output/` 裡同時有 `.pptx` 與 `review-report.md`。

### 10. 更新根目錄文件(INDEX.md / README.md / CLAUDE.md)

收尾時,在**專案根目錄**寫 / 刷新這三份文件,反映這次跑完的當前狀態。三者用途不同,各寫各的、不要互相複製貼上;每次跑都重寫成最新,不堆疊歷史(歷史看 `review-report.md` 與 `90_Excluded/`)。

**`INDEX.md` — 當前狀態地圖(給人快速掌握全局)**,建議涵蓋:
- **主題與狀態**:簡報主題、目前產到第幾版 / 產出檔名、最後更新時間。
- **OUTLINE 指紋與 SKILL 版本**:這次所用 `OUTLINE.md` 的指紋,以及本 skill frontmatter 的 `version`(供第 0 步下次比對是否變更)。
- **素材資料夾內容清單 / 指紋**:`20_References/` 與 `90_Excluded/`(排除素材)裡各資料夾目前的檔案清單(或內容指紋),供下次比對「資料夾有沒有被擴充 / 改動」(見第 1 步)。
- **設定**:套用的母片、長寬比。
- **章節地圖**:每章標題 → 審查小組(成員各是誰)→ 主要取自哪些素材(若該素材有 `URL.txt` 或上網查來的來源也標上)。
- **素材一覽**:`20_References/` 現有素材清單(標出 `(O)` 優先項);`10_Input/` 還有沒有待篩;`90_Excluded/` 排除了哪些 + 封存了哪些舊版;這次有沒有上網補資料、用了哪些網址。
- **產出位置**:`30_Output/` 下的檔案。

另外**存一份 `OUTLINE.md` 快照到 `30_Output/OUTLINE.used.md`**(覆蓋成這次用的版本),供第 0 步下次 diff 出「哪幾章變了」。

格式範例(精簡即可):

```
# 專案總覽:<主題>
更新時間:<YYYY-MM-DD HH:MM> · 目前版本:<簡報名>.pptx · 母片:<x> · 長寬比:16:9
OUTLINE 指紋:<hash 前 12 碼> · SKILL 版本:<version> · 審查小組:<列出成員代號>

## 章節地圖
1. <標題> — 完整小組通過 — 取自 <素材>(來源:<URL 若有>)
2. ...

## 素材一覽
- 20_References:<清單,標 (O)>
- 10_Input:<待篩數 / 已清空>
- 90_Excluded:排除 <n> 筆、封存舊版 <n> 份
- 上網補資料:<無 / 用了哪些網址>

## 產出
- 30_Output/<簡報名>.pptx、review-report.md、OUTLINE.used.md
```

**`README.md` — 給人看的專案說明**:這個專案是什麼主題、資料夾各區放什麼、素材該放哪(`20_References/` 或 `10_Input/`)、設定檔是根目錄的 `AUDIENCE.md` / `OUTLINE.md`、怎麼用 ref2pptx 重跑、產出在 `30_Output/`。大致穩定,主要確保存在且結構描述正確;有「目前版本」之類的狀態行就順手更新。

**`CLAUDE.md` — 給 Claude Code 的專案記憶**:讓未來在這個資料夾開的 Claude session 一讀就懂該怎麼接手。建議寫:這是一個 ref2pptx 專案、請依 ref2pptx skill 處理;資料夾結構與各區用途;控制檔是根目錄的 `AUDIENCE.md` / `OUTLINE.md`;參考素材只在 `20_References/`(別去根目錄找素材);產出規則(無巨集、進度條、來源網址、進度條上方);目前狀態(版本、母片、長寬比)。寫精簡、可執行的要點即可。

這三份都放根目錄、都不是素材;參考素材仍只放 `20_References/`。

## 範例

**範例 1 — 完整流程**
專案 `課程_GenAI測試/`:根目錄有 `AUDIENCE.md`,`10_Input/` 放了一堆剛蒐集的筆記與截圖,`20_References/` 已有核心規格,`00_Masters/` 有一個公司母片。
動作:先篩 `10_Input/` → 一張模糊到看不清的白板照、和一份過期版規格移到 `90_Excluded/` 並記原因,其餘移進 `20_References/` → 讀根目錄 `AUDIENCE.md` → 問使用者「套這個母片嗎 / 長寬比?」→ 用 `20_References/` 排章節、逐章寫 3 例、受眾審查 → 套母片、16:9、無巨集、加底部進度條產出 → `.pptx` 與 `review-report.md` 寫進 `30_Output/`。

**範例 2 — 沒有 AUDIENCE.md**
專案根目錄沒有 `AUDIENCE.md`。
動作:先用預設受眾庫提一組對應(哪章配哪位受眾)給使用者確認,確認後再往下;不要自己悶著頭假設。

**範例 3 — 想排除 20_References 裡的東西**
篩選時發現 `20_References/` 有一份明顯離題的舊文件。
動作:**先告知使用者**「這份看起來離題,建議排除」,得到同意再移到 `90_Excluded/`;不要逕自移走使用者刻意放的素材。

## 注意

- 素材篩選與受眾審查是**這份 skill 的兩個重點**,不要為了趕快產出而跳過。
- **排除 = 移到 `90_Excluded/`,絕不刪除**;每筆都要在 `EXCLUDED.md` 記原因,讓使用者能覆核。
- 例子的「貼近」比「數量」更重要;湊滿三個但全是抽象換句話說,等於沒過。
- 視覺與排版細節不在這份 skill 處理範圍,一律回到 `pptx` skill。
- 若執行環境沒有可用的 `pptx` skill(例如本機 Claude Code 未安裝文件 skill),改用通用方式(pptxgenjs / python-pptx)產出,但仍要套用本 skill 自己的規則:無巨集、指定長寬比、每頁進度條。
