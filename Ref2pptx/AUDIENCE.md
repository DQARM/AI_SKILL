# 受眾設定(AUDIENCE.md)

## 聽眾學歷基準(全體適用)
- 所有聽眾**都大學畢業**:一般知識、閱讀理解、邏輯推理是大學程度。
- **不要過度簡化 / 太一般化**:不用幼稚比喻、不把常識當新知解釋、不把聽眾當小白。
- 技術字眼以**高中就學過的為基準**:高中(含)以下的科學 / 數學 / 資訊名詞可直接用、不必解釋;**只有超過高中程度的專業術語才需要解釋**。
- 下方各人物誌的「不會 / 看不懂」是在這條底線**之上**再加各自的專業限制。

## 受眾人物誌

### PM-nontech — 完全沒有技術背景的 PM
- 背景:懂產品與流程,不懂程式、架構、指令。
- 日常工作:整理需求、排期、跨部門溝通、確認交付。
- 看了會卡住的東西:技術實作細節;英文技術縮寫**可接受**,但每張用到的投影片角落都要附全名(極小字),沒附全名才會卡住。

### PM-tech — 技術轉型 PM(原 RD / QA)
- 背景:技術出身轉做 PM,之前可能是 RD 或 QA,**有技術底、看得懂術語與技術內容**。
- 日常工作:接收外界來的資訊,消化後用**通識語言**講給其他人聽;或跟**客戶討論規格**。
- 看了會卡住的東西:術語本身不會卡他;他在意的是「**這段能不能被轉述成讓非技術的人 / 客戶也聽得懂**」——如果內容只有技術細節、缺了能對外講的白話框架或商業意義,他會覺得不好用。

### QA-nocode — 不會寫 code 的 QA
- 背景:懂測試設計與測試情境,但不會自己寫程式。
- 日常工作:設計測試案例、手動執行、記錄缺陷、確認驗收條件。
- 看了會卡住的東西:程式碼片段、CLI 指令、pipeline / CI 等開發流程術語。

### QA-guionly — 只會測 GUI、無法揣摩使用者的 QA
- 背景:會照測試案例一步步點 GUI(按鈕、表單、頁面)驗證,確認「畫面有沒有照規格動」;不會也不碰 GUI 底下的東西(API、資料庫、log、終端機、程式碼)。幾乎**無法自行推想「使用者到底想做什麼、會怎麼用」**。
- 日常工作:照既有測試案例手動點測,回報畫面與規格不符之處。
- 看了會卡住的東西:
  - 任何要繞過 GUI 的動作(查 log、打 API、看資料庫、下指令)——他做不到。
  - 任何要他「揣摩使用者意圖」「自己想像情境」的敘述——他想不出來。
  - 抽象指示如「站在使用者立場」「考慮使用者體驗」,沒有具體步驟他無從下手。
- 例子要貼近:把「使用者想達成什麼 → 在畫面上一步步怎麼點 → 預期看到什麼」寫到具體;情境(誰、想做什麼、在哪一頁)要直接寫明,不要留給他腦補。

### RD-spec2code — 只會把 spec 翻成 code 的 RD
- 背景:能照規格實作,較少做架構 / 取捨判斷。
- 日常工作:把需求 / 規格實作成程式。
- 看了會卡住的東西:高層架構取捨、跨系統設計假設、沒有對應到實作的空泛描述。

### USER-zen — 佛系終端用戶
- 背景:什麼都不懂,只負責用;遇到 bug 很自然就繞過,不發問、不抱怨。
- 審查重點:不會主動發問,看不懂就默默跳過——任何需要他主動動腦才跟得上的地方,他會無聲掉隊。
- 通過條件:被動讀也能跟到底,沒有需要自行推敲的斷點。
- 例子要貼近:直接照做就會的步驟,不假設他會腦補。

### USER-picky — 找碴終端用戶
- 背景:什麼都不懂;看不懂、不會用就直接說「這東西爛」。
- 審查重點:對含糊、跳步、術語零容忍,會挑出來罵——用來逼出內容最弱、最易誤解的點。
- 通過條件:找不到明顯可被挑成「看不懂 / 不會用」的地方(把最弱處補強到他無從挑剔)。
- 例子要貼近:清楚到挑不出毛病的具體例子。

## 審查小組(完整小組,全員都要過)
- PM = PM-nontech
- QA = QA-nocode(可改用 QA-guionly / QA-nolinux)
- RD = RD-spec2code
- USER-zen(佛系)
- USER-picky(找碴)
